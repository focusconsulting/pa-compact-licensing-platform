# ADR-0003: Users Table Schema and Auth Design Decisions

## Status

Accepted

## Date

2026-04-15

## Context

Adding authentication to the PA Compact Licensing Platform requires a `users` table that links
Cognito-authenticated identities to application roles and state scope. Several design choices
were needed for this table that will set conventions for the rest of the data model.

## Decision Drivers

- Single PostgreSQL database — no distributed ID generation requirement
- All tables need an audit trail (who created each row)
- Admin users are pre-seeded before their Cognito accounts are created
- Auth runs on every API request — performance characteristics matter
- Redis is already in the stack but should only be used where it adds clear value

## Considered Options and Decisions

### 1. Primary Key Type: `BIGSERIAL` vs `UUID`

#### Option A: `BIGSERIAL` (chosen)

8-byte auto-incrementing integer generated by the database sequence.

**Pros:**
- Smaller index size (8 bytes vs 16) — faster B-tree operations and joins
- Natural ordering — insert order is queryable without extra columns
- Simpler to work with in queries, logs, and debugging

**Cons:**
- Sequential — IDs are guessable if exposed in URLs

#### Option B: `UUID`

128-bit random identifier, typically `gen_random_uuid()`.

**Pros:**
- Non-guessable — safe to expose in public URLs
- Globally unique across databases — useful for distributed systems

**Cons:**
- 16 bytes — larger indexes, slower joins
- No natural ordering without a separate `created_at` sort
- No practical benefit for a single-database application with no sharding

**Decision:** `BIGSERIAL`. IDs are not exposed in public URLs; the sequential nature is not
a security concern for this application. `public_id UUID` (the Cognito `sub`) serves as the
externally-facing identifier where non-guessability matters.

---

### 2. Audit Trail: `created_by` on All Tables

All tables will carry a `created_by BIGINT NOT NULL REFERENCES users(id)` column to record
which user created each row.

#### Option A: Self-referential FK, `DEFERRABLE` (chosen)

Declare the FK as `DEFERRABLE` (initially immediate) so normal inserts enforce it immediately,
but seed transactions can explicitly defer it with `SET CONSTRAINTS ALL DEFERRED` to insert
the bootstrap admin rows before the FK is checked.

**Pros:**
- Constraint is enforced on every normal insert (data integrity by default)
- Seed script controls when deferral happens — no surprise deferred checks in production code

**Cons:**
- Seed script must remember to open a transaction and call `SET CONSTRAINTS ALL DEFERRED`

#### Option B: Allow `NULL` for bootstrap row

Make `created_by` nullable and allow the first admin to have `created_by = NULL`.

**Pros:**
- Simpler seed script

**Cons:**
- `NULL` in an audit column is semantically unclear and forces nullable handling everywhere
- Sets a bad precedent for the audit trail pattern

**Decision:** Option A. `DEFERRABLE` with an explicit `SET CONSTRAINTS ALL DEFERRED` in the
seed transaction is the right trade-off: strict in production, flexible for bootstrap.

---

### 3. Cognito Identity Column: `public_id UUID`

The Cognito `sub` claim is stored as `public_id UUID UNIQUE` rather than `TEXT`.

**Decision:** Cognito `sub` is always a UUID v4 — safe to assume and type accordingly. Stored
as `UUID` (not `TEXT`) for type safety and index efficiency. Named `public_id` (not
`cognito_sub`) because it serves as the externally-facing identifier for the user, decoupled
from the internal `BIGSERIAL` primary key. Nullable because rows are seeded before Cognito
accounts exist; backfilled on first login via `COALESCE` in the `UPDATE ... RETURNING` query.

---

### 4. Redis Caching for Auth Lookups

On every authenticated request, `get_current_user` runs:
```sql
UPDATE users SET public_id = COALESCE(...), ... WHERE email = $1 AND is_active = TRUE RETURNING ...
```

#### Option A: No caching (chosen)

Hit the database on every request.

**Pros:**
- Deactivating a user takes effect immediately
- No cache invalidation logic
- The indexed query is microseconds at this scale

**Cons:**
- One DB round-trip per authenticated request

#### Option B: Cache user record in Redis with TTL

Cache the `CurrentUser` result keyed by email with a short TTL (e.g. 5 minutes).

**Pros:**
- Eliminates DB query on repeated requests from the same user

**Cons:**
- Deactivated users remain active until TTL expires
- Cache invalidation needed on role/state changes
- Premature optimization for a low-traffic admin tool

**Decision:** No caching. The DB query is fast enough, and immediate deactivation is the
correct behavior for an access-control check. Revisit if profiling shows this as a bottleneck.

---

### 5. Role Storage: `TEXT` + `CHECK` constraint vs `ENUM` type

#### Option A: PostgreSQL `ENUM` type

```sql
CREATE TYPE user_role AS ENUM ('admin', 'state_staff', 'compact_admin', 'licensee');
```

**Pros:**
- Database-enforced type safety
- Slightly more compact on disk

**Cons:**
- Adding a new role requires `ALTER TYPE user_role ADD VALUE` — cannot be done inside a
  transaction in PostgreSQL (the new value is committed immediately, not rolled back on failure)
- This project uses **yoyo-migrations**, which wraps each migration file in a transaction by
  default. There is no simple way to opt out of that transaction for a single statement, so
  any migration that needs `ALTER TYPE ADD VALUE` would either fail outright or require
  restructuring the migration to run outside yoyo's transaction wrapper — adding friction
  every time a role is added
- Renaming or removing a value requires a full type rebuild
- The full set of roles is not yet known; locking them into a type now is premature

#### Option B: `TEXT` + `CHECK` constraint (chosen)

```sql
role TEXT NOT NULL,
CONSTRAINT chk_users_role CHECK (role IN ('admin', 'state_staff', 'compact_admin', 'licensee'))
```

**Pros:**
- Adding a new role is a standard `ALTER TABLE ... DROP CONSTRAINT / ADD CONSTRAINT` — fully
  transactional and reversible in a migration
- The allowed values are visible directly in the migration file without looking up a type
- No type object to manage across environments

**Cons:**
- No dedicated type name — slightly less self-documenting at the column level
- Marginally larger on disk than an enum (negligible)

**Decision:** `TEXT` + `CHECK` constraint. The full set of roles is not yet settled, and
transactional migrations matter more than the minor ergonomic benefit of a named enum type.

---

## Consequences

### Positive

- `BIGSERIAL` PKs keep indexes lean and joins simple for all future tables
- `created_by` on all tables gives a full audit trail from day one
- `public_id` cleanly separates internal DB identity from Cognito identity
- Deactivating a user takes effect on the next request with no cache to flush
- New roles can be added in a fully transactional migration without type rebuilds

### Negative

- Seed scripts for any table with `created_by` must use a transaction with
  `SET CONSTRAINTS ALL DEFERRED` — Mitigation: document this pattern clearly in seed file comments
- `public_id` is `NULL` on seeded rows until first login — Mitigation: `CurrentUser.public_id`
  is typed `UUID | None`; callers must handle `None` until the user has logged in at least once
- `role` has no dedicated DB type — Mitigation: the `CHECK` constraint is the source of truth;
  the allowed values are visible in the migration file
- `given_name` and `family_name` are nullable and populated opportunistically from token claims.
  If additional IdPs are introduced (Azure AD, SAML providers, Apple Sign In), these fields
  cannot be relied upon — IdP attribute mapping is inconsistent, and some providers (notably
  Apple) only send name claims on the first login. If a reliably present display name is needed
  in the future, a `display_name TEXT` column set explicitly at provisioning time should be
  added rather than deriving it from these fields.
